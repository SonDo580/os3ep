# Scheduling metrics

- **Turnaround time:**

```
T_turn_around = T_completion - T_arrival
```

- **Response time:**

```
T_response = T_first_run - T_arrival
```

# Workload assumptions (unrealistic)

1. Each job runs for the same amount of time
2. All jobs arrive at the same time
3. Once started, each jobs run to completion
4. All jobs only use the CPU (perform no I/O)
5. The run time of each job is known

# Scheduling policies

## First in, first out (FIFO) or First come, first served (FCFS)

- **Relax:**
  - assumption #1 -> jobs don't run in the same amount of time
  - But let jobs arrive just a hair before each other
- **Bad scenario:**
  - heavier jobs come first: increase both average `T_turn_around` and average `T_response` (compared to when lighter jobs come first)

## Shortest job first (SJF)

- **Relax:**
  - assumption #1 -> jobs don't run in the same amount of time
- **Bad scenario:**
  - Relax assumption #2 -> jobs don't arrive at the same time -> if a heavy job comes first, it runs to completion, even when shorter jobs arrive shortly after.

## Shortest time-to-complete first (STCF) or Preemptive shortest job first (PSJF)

- **Relax:**
  - assumption #1 -> jobs don't run in the same amount of time
  - assumption #2 -> jobs don't need to arrive at the same time
  - assumption #3 -> jobs don't need to run to completion _(in 1 run)_
- **Idea:** Any time a new job enters the system, the scheduler determines which of the remaining jobs (including the new one) has the least time left, and schedule that one.
- **Good for** `T_turn_around`
- **Bad case:**
  - If multiple jobs arrive at the same time, the longest job must wait for the others to run to completion -> bad for `T_response` (and interactivity)

## Round Robin (RR)

- **Relax:**
  - assumption #1 -> jobs don't run in the same amount of time
  - assumption #2 -> jobs don't need to arrive at the same time
  - assumption #3 -> jobs don't need to run to completion _(in 1 run)_
  - assumption #5 -> don't need to know the run time of each job in advance
- **Idea:** Instead of running jobs to completion, run a job for a **time slice** (sometimes called a **scheduling quantum**), then switch to the next job in the queue. Repeat until all jobs are finished.
- **Good for** `T_response`
- **Note:**
  - The length of a time slice must be a multiple of the timer-interrupt period.
  - The shorter the time slice, the better `T_response`. However, making it too short is problematic: **context switching** cost will dominate.
- **Bad case:**
  - Bad for `T_turn_around`: RR stretches out each job, but turnaround time only cares about when jobs finish (any **fair** policy will perform poorly on `T_turn_around`)

# Incorporating I/O

- **Relax:**
  - assumption #1 -> jobs don't run in the same amount of time
  - assumption #2 -> jobs don't need to arrive at the same time
  - assumption #3 -> jobs don't need to run to completion _(in 1 run)_
  - assumption #4 -> jobs can perform I/O
- **Idea (with STCF):** Treat each CPU burst as a separate job. While the interactive jobs are performing I/O, other CPU-intensive jobs run.
- **Example:**
  - Assume we have 2 jobs: A and B. B uses the CPU for 50ms and performs no I/O. A runs for 10ms then issues an I/O request (assume each I/O takes 10ms).
  - Treat each 10ms sub-job of A as an independent job.
  - When the system starts, it chooses the 1st sub-job of A. When the 1st job of A has completed, only B is left, and it begins running. Then a new sub-job of A is submitted, and it preempts B and run for 10ms.
  - Doing so allows **overlap**. The CPU is used by 1 process while waiting for I/O of another process to complete.
