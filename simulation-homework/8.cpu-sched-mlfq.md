# 4: Craft a workload with 2 jobs and scheduler parameters so that 1 job takes advantage of the older Rules 4a and 4b (turned on with the -S flag) to game the scheduler and obtain 99% of the CPU over a particular time interval.

- To game the scheduler, job 0 will issue an I/O request right before the allotment is used up, job 1 will not issue I/O requests.
- Need at least 2 queues, so that job 1 can only run 1 full allotment while in the top queue, then be moved to the lower-priority queue.
- Workload:

```bash
./mlfq.py -S -n 2 -q 100 -a 1 -i 1 -l 0,300,99:0,300,0
```

# 6: One question that arises in scheduling is which end of a queue to add a job that just finished I/O; the -I flag changes this behavior for this scheduling simulator. Play around with some workloads and see if you can see the effect of this flag

```bash
./mlfq.py -s 10 -n 2 -q 5 -a 1 -j 2 -m 50 -M 10 -i 1 -I

# OPTIONS jobs 2
# OPTIONS queues 2
# OPTIONS allotments for queue  1 is   1
# OPTIONS quantum length for queue  1 is   5
# OPTIONS allotments for queue  0 is   1
# OPTIONS quantum length for queue  0 is   5
# OPTIONS boost 0
# OPTIONS ioTime 1
# OPTIONS stayAfterIO False
# OPTIONS iobump True

# Job List:
#   Job  0: startTime   0 - runTime  28 - ioFreq   4
#   Job  1: startTime   0 - runTime  29 - ioFreq   2
```

- **Effect:** When an I/O completes, the job that issued it gets run immediately (if its allotment still remains).
